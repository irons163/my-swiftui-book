# 附錄 B Xcode 與 SwiftUI 專案基本操作

## 這份附錄的定位

這份附錄不是完整的 Xcode 教學，而是替本書正文補齊最常用、也最容易在一開始卡住的幾個基本操作。

如果你已經會：

- 建立專案
- 跑 Preview
- 開 Simulator
- 找基本錯誤訊息

那你可以快速略過。

## 1. 建立一個 SwiftUI 專案

在 Xcode 中建立新專案時，你最常會碰到這幾個欄位：

- Product Name：專案名稱
- Team：若要簽署或上實機時會用到
- Organization Identifier：反向網域名稱
- Interface：選 `SwiftUI`
- Language：選 `Swift`

對這本書來說，最重要的是後兩個：

- `Interface = SwiftUI`
- `Language = Swift`

只要這兩個對，正文中的大部分操作才能直接接起來。

## 2. 你會一直看到的幾個區域

一開始不用想把整個 Xcode 介面全部搞懂，先認識最常用的幾塊：

- 左側 Navigator：看專案檔案
- 中央 Editor：寫程式碼
- 右側 Inspector：看檔案或元件設定
- 上方工具列：執行 App、切換裝置
- 下方 Console：看錯誤、輸出和執行訊息

只要這幾塊開始熟，正文的跟做成本就會低很多。

## 3. 怎麼跑 Preview

SwiftUI 裡最常用的即時回饋工具之一就是 Preview。

正文裡你會一直遇到：

```swift
#Preview {
    HabitCheckInCard()
}
```

如果 Preview 沒有自動顯示，你可以先檢查：

- 是否已打開 Canvas
- 程式碼是否有明顯錯誤
- `#Preview` 區塊是否完整

一開始你不需要知道 Preview 背後所有機制，只要先把它當成：

`快速看畫面目前長什麼樣子的地方。`

## 4. 怎麼跑 Simulator

當你需要檢查完整互動流程時，就要跑模擬器。

最基本的流程是：

1. 在 Xcode 上方選一台模擬裝置。
2. 按執行按鈕。
3. 等待 Simulator 開啟並安裝 App。

Preview 適合快看畫面狀態，Simulator 比較適合：

- 檢查整條操作流程
- 檢查導航與手勢
- 檢查多畫面切換
- 看真正執行時的感覺

## 5. 正文裡最常見的檔案類型

你在這本書裡會最常碰到三種：

- `App` 入口檔
- `View` 檔
- 資料模型或功能模型檔

可以先這樣理解：

- `App`：整個程式從哪裡開始
- `View`：畫面長什麼樣子
- Model / Store / Repository：資料和邏輯放在哪裡

這樣一來，正文談架構時你比較不會完全失去方向。

## 6. 錯誤訊息先看哪裡

很多初學者不是不會修錯，而是不知道第一眼該看哪裡。

建議你先看這三件事：

1. 紅色錯誤停在哪一行。
2. 錯誤訊息裡提到的是哪個型別、哪個名稱、哪個參數。
3. 這個錯誤是語法錯，還是資料型別對不上。

最常見的幾種錯誤類型通常是：

- 少了括號或大括號
- 名稱打錯
- 傳進去的型別不對
- 某個值原本該是 `Binding`，你卻傳成一般值

如果你先分得出這幾類，很多錯誤就不會看起來像天書。

## 7. 這本書裡你至少要會的 Xcode 操作

讀正文前，建議你至少要能做到：

- 建立一個 SwiftUI 專案
- 新增一個 Swift 檔或 SwiftUI View 檔
- 跑 Preview
- 跑 Simulator
- 看懂哪裡有紅色錯誤
- 知道 Console 在哪裡

這樣就夠順著本書一路往下走。

## 8. 一開始最容易卡住的地方

### Preview 沒反應

先檢查是不是程式碼本身有錯，因為只要有編譯錯誤，Preview 往往也會一起失效。

### App 跑不起來

先不要立刻懷疑整個專案壞掉。很多時候只是：

- 某行語法寫錯
- 有型別對不上
- 某個 closure 沒有正確結束

### 視窗太多，不知道自己在哪

不用一次理解整個 IDE。先專注在：

- 檔案在哪裡
- 程式碼寫在哪裡
- 錯誤顯示在哪裡
- App 怎麼跑起來

這四件事穩了，其他會慢慢跟上。

## 本附錄小結

這本書真正要教的不是 Xcode 本身，但如果你連最基本的操作都還卡住，正文的學習成本會被放大很多。所以這份附錄的目標很單純：先讓你在工具層不要跌倒，才有力氣把注意力放回 SwiftUI 本身。
